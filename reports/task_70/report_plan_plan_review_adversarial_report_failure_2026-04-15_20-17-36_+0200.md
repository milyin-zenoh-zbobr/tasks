Plan v22 is **not implementation-ready yet**. It fixes the v21 fast-forward problem, but two blocking design issues remain after checking the actual code paths and the issue contract.

1. **Post-rewrite `collect_merge_commits(pre_update_head..HEAD)` is not stable after `filter-branch`.**
   - The plan captures `pre_update_head`, runs `update_worktree`, then (when `overwrite_author` is enabled) runs `rewrite_authors_on_worktree(work_dir, commit_baseline, &pre_commits, ...)`, and after that proposes to re-collect sync commits with `collect_merge_commits(work_dir, &pre_update_head)`.
   - In the current utility, `rewrite_authors_on_worktree` is implemented with `git filter-branch ... '<lower_bound>..HEAD'` (`zbobr-utility/src/lib.rs:327-371`). Even if the env-filter only changes the selected agent commits, `filter-branch` rewrites descendants too, so the first-parent chain above any rewritten stage commit gets new SHAs.
   - In the dispatcher flow, `pre_update_head` is specifically the pre-sync HEAD of the stage (`zbobr-dispatcher/src/cli.rs:2115-2189` is the current stash/update/rewrite path the plan is modifying). That commit is typically one of the stage commits being rewritten. After rewrite, that old hash is no longer the correct boundary on the rewritten history.
   - As a result, `collect_merge_commits(pre_update_head..HEAD)` after rewrite can no longer be relied on to mean “merge commits created by the sync step of this stage.” It may over-collect earlier first-parent merge commits from the rewritten branch history, because the exclusion side of the range is anchored to a pre-rewrite commit graph.
   - **Required fix:** the design must stop depending on a pre-rewrite SHA as the post-rewrite merge-collection boundary. The plan needs a rewrite-safe way to carry sync-created commits forward — for example, a design that identifies/map those sync commits across the rewrite, or a different post-rewrite collection strategy whose lower bound is guaranteed to exist on the rewritten history.

2. **The plan still does not satisfy the issue’s explicit “short hashes” requirement for the user-visible context.**
   - The issue says: “append section in the stage record in the context with **short** PR hashes made during this stage. This section should be available both to user and prompt” (`issue #314`).
   - But v22 explicitly says: persist-mode emits **full SHAs**, prompt-mode emits 12-char abbreviations.
   - In this codebase, the persisted/user-visible task context is exactly the `serialize_context(..., false, ...)` output, and the GitHub backend parses that same markdown back with `parse_context(...)` (`zbobr-task-backend-github/src/separator.rs:97, 141`). So choosing full SHAs for persist-mode is not an internal-only representation choice — it directly changes what the user sees in the task context.
   - That means the plan currently implements “full hashes for users, short hashes for prompts,” which contradicts the issue requirement “short hashes ... available both to user and prompt.”
   - **Required fix:** the plan needs a representation that preserves whatever data `overwrite_author` needs **without** exposing full hashes in the user-visible context. That could mean storing full hashes separately from the rendered markdown, or intentionally storing only abbreviations and defining a safe prefix-resolution rule for later rewrite logic. But the current “full in persist, short in prompt” split is not acceptable.

With those two points addressed, the rest of the plan remains directionally aligned with the repository structure and the previously identified problem areas.
